## Decision

The in-app onboarding form drops `general_availability` and `preferred_venues`.
Both net-new paths (slot-context and generic) now use **one form**: `first_name`,
`last_name`, `skill_level`, optional gender.

D7.2's slot-vs-generic field split is retired. The only remaining per-flow
difference is **post-submit destination**:

| Flow | Path |
|------|------|
| Slot-context (Flow 2) | `signup_claim` + `join_slot` → lobby |
| Generic (Flow 1) | `signup_claim` → home |
| Claim (Flow 3) | still skips the form |
| Returning (Flow 2′ / returning) | still skips the form |

---

## Rationale

v1 does no venue- or time-based matching — everyone sees all games.
`general_availability` and `preferred_venues` have **zero v1 read path**:
no query, no filter, no display, no decision uses them. Asking for them is
friction without value. This is the same test that cut `willing_to_drive` in
D7.1. Collapsing to one form also reduces 3B to a single component; the D2
branch only picks the post-submit destination.

---

## Schema

No migration required. Both columns already exist and are nullable (established
by D7.2's downstream commits). They remain NULL for all in-app signups going
forward and stay as the open seam for a future optional profile surface.

---

## Waitlist asymmetry (acceptable)

Flow 3 claim users already have `general_availability` and `preferred_venues`
populated from the Waitlist Intake Script. In-app signups will have them NULL.
v1 uses neither field for any read path, so the asymmetry is inert.

---

## Deferral

`general_availability` and `preferred_venues` move to a **future optional profile
surface** — returning user, at their own pace, never onboarding friction. Trigger:
when v1 ships venue- or time-aware matching. Breadcrumb in v2-signals.md.

---

## Copy update (D2 register)

The single post-verify line is replaced per flow (proposed; lock when 3B ships):

- Slot-context: `phone verified. you're one step from the game.`
- Generic: `phone verified. let's find your game.`

Supersedes `complete your profile to secure your spot.` for these surfaces.

---

## Does NOT change

D7 core mechanics, group 6, curation, waitlist · D2 flows and branching logic ·
D7.3 optional gender field · D10 / Phase-2 pre-join privacy posture (abstract
count via `slot_share_preview`, never member identities) · name split into
`first_name` / `last_name` (D7.2 / D3 identity-resolution, still required).
