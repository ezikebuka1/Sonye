# D7.4 — Onboarding Form Collapse

Status: **Locked** (flipped from PENDING LOCK at Phase 3B implementation, 2026-06-10; originally drafted 2026-06-09, repo commit 85fecc0)
Amends: D7 (builds on D7.1); supersedes D7.2's field bifurcation
Triggered by: 3B form-sketch review — both remaining asked-but-unused
fields failed the exact test that cut willing_to_drive in D7.1.

## Decision

The in-app onboarding form for ALL net-new users — both entry
contexts — collapses to ONE unified form with four fields:

- first_name (required)
- last_name (required)
- skill_level (required; 4-tier radiogroup)
- gender (optional; radiogroup, skippable — D7.3 unchanged)

`general_availability` and `preferred_venues` are dropped from
in-app onboarding entirely. Recorded NULL for all in-app signups.

D7.2's slot-vs-generic FIELD bifurcation is retired. The two
net-new flows (Flow 1 generic, Flow 2 slot-context) now differ
only in frame and post-submit behavior, never in fields:

| | Flow 2 (slot) | Flow 1 (generic) |
|---|---|---|
| Banner | pinned "You're joining…" (+ "N already in" if fill ≥50%) | none |
| Header | "phone verified. you're one step from the game." | "phone verified. let's find your game." |
| CTA | "Join the game" | "Find games" |
| Post-submit | `signup_claim` → best-effort `join_slot` → lobby/home | `signup_claim` → home |
| Footer | D10 phone disclosure under CTA | "your number stays private until you join a game" |

Paths unchanged: Flow 3 (waitlist claim) and returning users see
no form at all.

## Why

Same test as D7.1 (willing_to_drive): a field that no v1 read
path consumes is pure friction. Neither `general_availability`
nor `preferred_venues` is read anywhere in v1 — there are only
three vetted venues, and slot opt-in IS the availability signal.
Every field cut from the top-of-funnel form is a density
multiplier (Partiful Model); every retained-but-unused field is
a density tax.

D7.2's insight (the join action is stronger data than a
self-reported form) was right but didn't go far enough: it cut
the fields only for the slot path. The generic path deserves the
same cut for the same reason.

## Schema implications

None. Both columns are already nullable (D7.2). They stay in the
schema as v2 seams (future profile/matching surface). No
migration.

## Locked copy (D5 register)

- First-name error: *"we need a first name — it's how your group
  knows you"*
- Skill helper: *"pick whatever feels right — you can change it
  later"*
- Headers/CTAs/footers: per the table above. Gemini-locked.

## Accessibility (adopted at 3B; corrected during verification)

- Touch targets: 46px chips / 52px inputs / 56px CTA
- Explicit `<label for>` on every input
- Skill AND gender are **radiogroups**: `role="radiogroup"` +
  `role="radio"` + `aria-checked` on each chip. (The original
  D7.4 board specified `aria-pressed`; corrected at 3B
  verification — these are single-select, not toggles. Gender's
  "none selected" default is a first-class state.)
- Selection never color-alone: tint + ring + ✓
- Errors: `role="alert"` + `aria-invalid="true"` + resolving
  `aria-describedby`; error red `#D64B4B`; validate on submit,
  never while typing; on failed submit scroll to and focus the
  first error

## Does NOT change

D7 core mechanics, D7.1 (willing_to_drive), D7.3 (optional
gender), the D2 flow structure (1 / 2 / 2′ / 3), phone via OTP,
the Waitlist Intake Script's full 7-field manual collection
(pre-launch waitlist rows still arrive fully populated).
