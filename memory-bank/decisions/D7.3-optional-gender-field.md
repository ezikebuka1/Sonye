# D7.3 — Optional Gender Field

Status: Locked 2026-05-16
Amends: D7
Triggered by: Real-user feedback (women players) — gender composition
matters for safety/comfort in casual mixed sport.

## Decision

Onboarding adds an OPTIONAL gender field. Options:
Woman / Man / Non-binary / Prefer not to say. Skipping = same as
"prefer not to say".

Used to derive avatar color (see D8.1, amended by D8.1a):
- Woman → pink family
- Man → blue family
- Non-binary → green; Prefer not to say / NULL → blue (default neutral)

"Skipping = same as 'prefer not to say'" is PRESERVED: a skip (stored
NULL) and an explicit "prefer not to say" now BOTH resolve to blue, so
the opt-out stays indistinguishable. (Amended 2026-06-14; was "→ green".)

Optional because forcing the question adds onboarding friction and can
feel invasive — contradicts Density Over Everything. NULL is a
first-class state, never an error.

## Visibility

Avatar colors are member-context only. Visible in the lobby AFTER a user
joins a game. NOT shown on the public/anonymous share preview. NOT shown
on home-screen slot cards before joining. The strong safety mechanism is
the slot gender tag (see D3 gender-category amendment), not the avatar
color — the tag is a commitment; color is a changeable snapshot.

Enforced in code (2026-06-14): the Home fill dots carry NO gender —
they render a single neutral (#AEBED0); gender→color derivation
(`getAvatar` in `src/lib/avatar.ts`) is reached only by the lobby roster.

## Schema implications (feed D3)

- users.gender TEXT NULL
  CHECK (gender IN ('woman','man','non_binary','prefer_not_to_say')
         OR gender IS NULL)
- Avatar color derived from gender at render time, not stored

## Does NOT change

Skill, name, phone, all other onboarding. Purely additive.
